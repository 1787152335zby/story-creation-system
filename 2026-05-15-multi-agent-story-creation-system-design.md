## 一、系统定位

### 1.1 核心理念

本系统是一个**通用的多智能体故事创作系统**，让用户以"超级个体"的身份，管理一群AI Agent，完成从创意到视频的完整创作流程。

### 1.2 关键原则

- **通用性**：不局限于短剧，支持电影、电视剧、小说、舞台剧、广播剧等任何故事形式
- **渐进式创作**：先搭骨架（大纲）→再填肉（剧情）→再细化（分镜）→再生产（视频）
- **人机协作**：用户是决策者+审核者，Agent是执行者
- **一键便携**：纯Python项目，文件夹一拷就走，无需复杂环境配置

### 2.2 两种工作模式

| 模式 | 说明 | 适用场景 |
|:----:|:------|:---------|
| **全自动链式** | 用户提出创意，系统自动串联全部Agent，完成从大纲→剧情→分镜→提示词→视频 | 用户信任系统，希望快速出结果 |
| **分阶段审核** | 每个阶段产出后暂停，等用户审核确认后再继续下一阶段 | 用户需要把控创作方向 |

**默认采用分阶段审核模式**，用户可在运行中选择是否进入全自动模式。

### 2.3 可配置工作流（核心扩展性设计）

总指挥官的执行流程不硬编码在代码中，而是通过 **workflow.yaml** 配置文件定义。

**这样设计的好处：**
- 增加新Agent → 只需在yaml中加一行
- 调整执行顺序 → 修改yaml即可
- 条件执行 → 某些阶段只在特定故事类型下启用
- 无需修改总指挥官的代码

**workflow.yaml 示例：**
```yaml
phases:
  - name: 故事大纲
    agent: outline_designer
    output: 01_故事大纲.md
    condition: true

  - name: 完整剧情
    agent: plot_expander
    output: 02_完整剧情.md
    condition: true

  - name: 分镜设计
    agent: storyboarder
    output: 03_分镜脚本.md
    condition: "story_type in ['短剧', '电影', '电视剧']"

  - name: 视频提示词
    agent: prompt_engineer
    output: 04_提示词.md
    condition: "story_type in ['短剧', '电影', '电视剧']"

  - name: 视频生成
    agent: video_producer
    output: 05_视频/
    condition: "story_type in ['短剧', '电影', '电视剧']"
```

未来如果需要增加新的Agent（如"配音脚本师"），只需：
```yaml
  - name: 配音脚本
    agent: voice_script_engineer
    output: 06_配音脚本.md
    condition: "story_type in ['短剧', '电影', '电视剧', '广播剧']"
```

## 三、Agent 详细设计

### 3.1 🧠 总指挥官（Orchestrator）

| 维度 | 设计 |
|:----:|:------|
| **角色定位** | 执行制片人 / 项目经理 |
| **触发方式** | 用户直接对话，或通过文件下达任务 |
| **核心能力** | 理解意图 → 拆解子任务 → 分配Agent → 汇总结果 → 汇报给用户 |

**工作流程：**
```
用户："帮我创作一个故事"
  ↓
总指挥官：分析任务
  ↓ 第一步：询问故事类型（见第3.2节）
  ↓ 第二步：根据类型询问风格/题材
  ↓ 第三步：调用大纲设计师
  ↓ 第四步：让用户审核大纲
  ↓ 第五步：通过后调用剧情展开师
  ↓ 第六步：让用户审核剧情
  ↓ 第七步：通过后调用分镜师（仅视频类故事）
  ↓ 第八步：让用户审核分镜
  ↓ 第九步：通过后调用提示词工程师（仅AI视频类）
  ↓ 第十步：让用户审核提示词
  ↓ 第十一步：调用视频Agent（仅当需要视频生成时）
  ↓ 第十二步：让用户审核最终视频
```

### 3.2 📐 大纲设计师（Story Outline Designer）

**核心理念 — 风格金字塔：风格不是在一个阶段定义的，而是层层叠加的**

```
                   定义什么
                        ▲
   分镜/提示词  ──────  画面质感层（光影/镜头/色调/运镜）
                        │
         剧本  ──────  叙事节奏层（对白节奏/场景长度/悬念密度）
                        │
         大纲  ──────  结构骨架层（三幕/英雄之旅/高潮节点）
                        │
                        风格选择
```

风格在**大纲阶段选定**，然后像一个配置文件一样沿着流水线一路传递，每一层Agent都在自己层面做对应适配。

**启动时的必问流程：**

```
第一步：询问故事类型
  选项：
    ├─ 🎬 短剧（竖屏，每集1-3分钟，适合AI视频生成）
    ├─ 🎥 电影（90-150分钟，完整三幕结构）
    ├─ 📺 电视剧（每集45分钟，多集/多季）
    ├─ 📖 小说/网文（纯文字，按章节）
    ├─ 🎭 舞台剧/话剧（幕场结构）
    ├─ 🎙️ 广播剧/有声书（声音驱动，无画面）
    └─ ✏️ 其他（用户自定义）

第二步：询问题材风格（根据所选类型动态生成选项）
    短剧示例：都市情感 / 悬疑反转 / 古装虐恋 / 喜剧 / 战神归来 / ...
    电影示例：科幻 / 悬疑推理 / 动作 / 爱情 / 历史 / 动画 / ...
    小说示例：玄幻 / 修仙 / 都市 / 言情 / 科幻 / 灵异 / ...

第三步：询问文笔风格
  📖 请选择你期望的文笔风格（影响剧本/剧情阶段的文字质感）：
    1) ✨ 精炼实用（干净利落，适合短剧/快节奏）
    2) 🎨 文学质感（细腻生动，适合小说/文艺片）
    3) 🎭 对白优先（对话驱动，适合舞台剧/广播剧）
    4) 🌟 画面感强（视觉化描写，适合电影）
    5) 🤖 自动适配（根据故事类型自动选择最优风格）

第四步：询问视觉/叙事风格
  🎬 请选择期望的视觉/叙事风格（影响分镜/提示词的风格取向，
     可多选则混合风格）：
    1) 🎥 好莱坞大片风（三幕结构、视觉冲击、电影感镜头语言）
    2) 📱 竖屏短剧风（快节奏、强反转、情绪密集）
    3) 🎨 文艺/独立风（慢节奏、长镜头、情绪留白）
    4) 🏠 日韩生活风（细腻日常、温暖治愈、轻节奏）
    5) 📺 电视剧风（多线叙事、人物群像）
    6) ✏️ 自定义（用户自己描述）

第五步：询问画面比例
  📏 请选择目标画面比例（影响分镜和提示词的构图描述）：
    1) 📱 9:16 竖屏（短剧常用，强调人物近景/纵深）
    2) 🖥️ 16:9 横屏（电影/电视剧标准，强调横向调度）
    3) 🤖 自适应（让Agent根据故事类型自动选择）

第六步：询问剧本写作风格（仅视频类故事类型需要）
  🎭 请选择剧本写作风格（影响剧本中对白与动作描写的比例）：
    1) 🎬 视觉化写作（动作描写先行，对白精炼，"show don't tell"）
    2) 🗣️ 对白驱动型（大量对白驱动剧情，轻描写重对话）
    3) 📖 文学剧本型（细腻动作/心理描写+对白，文体接近文学剧本）
    4) ✏️ 自定义（你描述想要的写作风格）

第七步：询问具体要求和约束
  - 主题方向（如"女性成长"、"正义必胜"）
  - 情绪基调（如"憋屈→逆袭"、"温暖治愈"）
  - 大致篇幅（短剧多少集、电影多长、小说多少字）
  - 参考作品（如有）
  - 禁忌/合规要求

第八步：生成两版大纲——"版本A（经典结构）"和"版本B（创新/反套路）"

### 3.3 📖 剧情展开师（剧本创作 Agent）

| 维度 | 设计 |
|:----:|:------|
| **角色定位** | 专业剧本创作大师 |
| **输入** | 审核通过的故事大纲 + 风格配置 |
| **输出** | 完整剧本（带对白、动作描写、情绪标注） |
| **核心原则** | 整体连贯、人物鲜活、情绪可见、避免长对话混乱 |

#### 3.3.1 工作流程

```
大纲审核通过
      ↓
读取风格配置（文笔风格 + 剧本写作风格 + 画面比例）
      ↓
按集/按幕/按场展开剧本
      ↓
每场包含：场景设定 → 动作描写 → 情绪标注 → 对白
      ↓
输出完整剧本文件 → 你审核 → 修改 → 通过
```

#### 3.3.2 人物刻画方式

采用**混合式**——动作描写为主，关键情绪加括号标注：

```
林向晚手指绞着衣角，指尖泛白。
          （紧张，强装镇定）
"我没事……真的。"
她扯出一个笑容，但笑意没到眼底。
```

| 要素 | 作用 | 示例 |
|:----:|:------|:------|
| **动作描写** | 让读者"看到"角色状态，有画面感 | 手指绞着衣角，指尖泛白 |
| **情绪标注** | 让AI/演员准确理解情绪 | （紧张，强装镇定） |
| **对白** | 推进剧情，展现人物性格 | "我没事……真的。" |
| **反应描写** | 展现对白后的情绪余波 | 笑意没到眼底 |

#### 3.3.3 剧本写作风格

| 风格 | 对白占比 | 动作描写 | 适合类型 |
|:----:|:--------:|:---------|:---------|
| **🎬 视觉化写作** | ~40% | 详细，画面感强 | 悬疑/动作/电影感强的故事 |
| **🗣️ 对白驱动型** | ~65% | 简洁，只写关键动作 | 情感剧/家庭伦理/舞台剧 |
| **📖 文学剧本型** | ~50% | 细腻，含心理描写 | 小说改编/文艺片 |

#### 3.3.4 画面比例对剧本的影响

| 比例 | 场景描写侧重点 |
|:----:|:--------------|
| **9:16 竖屏** | 偏人物近景、纵深描写、上半身动作、垂直调度 |
| **16:9 横屏** | 偏环境关系、左右调度、全身动作、多人同框 |

#### 3.3.5 输出格式（完整版）

**短剧格式示例：**
```markdown
# 第1集
> 写作风格：视觉化写作 | 画面比例：9:16 竖屏
> 情绪定位：压抑、憋屈 | 本集时长：约2分30秒

---

#### 3.4.1 场景参考库（核心机制）

场景参考库是连接「抽象设定」和「具体画面」的桥梁。它在分镜阶段之前自动生成。

**每个核心场景生成一个参考文件：**
```
项目目录/
├── 场景设定/
│   ├── 林家客厅_v1.md
│   ├── 公司办公室_v1.md
│   ├── 方晴家_v1.md
│   └── (以此类推，每个高频场景一个文件)
```

**场景参考文件内容示例：**
```markdown

|           上层常有未收拾的杯子 |

## 🔑 机位-描述映射表
| 机位 | 拍摄方向 | 画面描述 |
|:----:|:---------|:---------|
| **A位-沙发侧** | 从左侧沙发→面向中央茶几和电视柜 | 前景沙发虚化，中景茶几，背景电视柜和落地窗 |
| **B位-电视柜侧** | 从右侧电视柜→面向中央茶几和沙发 | 前景绿萝虚化，中景茶几，背景沙发和落地窗 |
| **C位-门口** | 从门口→正对落地窗全景 | 三人同框，完整空间展露 |
| **D位-茶几上方** | 俯拍→向下拍茶几 | 俯视茶几上的物品，展示细节 |
| **E位-人物近景** | 从正面或侧面拍人物胸部以上 | 背景虚化的沙发或墙壁 |

#### 3.4.2 场景状态链

每个镜头生成时，自动维护一张状态变化表，确保AI知道场景的**当前状态**。

**状态链规则：**
```
┌────────────────────────────────────────────┐
│ 每个镜头结束 → 记录状态变更                  │
│ 下一个镜头 → 读取上一个镜头的结束状态          │
│ 状态变更类型：                              │
│   ├─ 物品状态：杯子完好→碎了                  │
│   ├─ 人物状态：衣服干净→湿了/脏了             │
│   ├─ 场景布局：椅子整齐→倒了                  │
│   ├─ 光影变化：白天→关灯变暗                  │
│   └─ 情绪氛围：平静→紧张                     │
└────────────────────────────────────────────┘
```

**状态链在分镜中的体现：**
```markdown
#### 🎬 镜头001 | 3s
**状态（初始）：客厅整洁，茶几上有完好玻璃杯**
...
→ 结束状态：玻璃杯被摔碎，碎片在地上，水渍蔓延

#### 🎬 镜头002 | 4s
**状态（继承）：地上有玻璃碎片和水渍**
...
→ 结束状态：碎片被踢散，地面上有零星血迹
```

#### 3.4.3 场景版本升级机制

当场景发生**结构性破坏**时（如爆炸、坍塌、火灾），自动触发场景版本升级：

```markdown
#### 🎬 镜头015 | 5s
**房子开始坍塌**
场景状态：林家客厅 v1 → 破坏中

#### 🎬 镜头016 | 3s
**💡 建议：加入黑场过渡（1-2秒），提示场景发生根本变化**
🏗️ 场景版本：林家客厅 v1 → 林家客厅·废墟 v2
状态链已重置（结构性破坏不再继承原有物品位置）
```

**版本升级自动处理流程：**
```
1. 分镜师识别"破坏程度超过阈值"
   → 承重墙倒塌/天花板塌陷/大面积结构损毁
2. 自动生成v2场景设定（无需用户手动）
   → 保留固定地标方位逻辑，但内容变为废墟状态
3. 场景版本号+1，状态链重置
4. 建议插入转场镜头（黑场/白场/闪白）
```

#### 3.6.4 API调用层设计

**主流程（以Seedance为例）：**
```
API调用请求
  ↓
① 构建请求参数

  ↓
② 提交任务
   POST https://api.seedance.com/v1/video/generate
   → 返回 task_id
  ↓
③ 轮询状态（每30秒）
   GET https://api.seedance.com/v1/video/status/{task_id}
   → 返回: queued / processing / completed / failed
  ↓
④ 完成 → 下载视频
   GET https://api.seedance.com/v1/video/{task_id}/download
   → 保存到本地
  ↓
⑤ 更新状态文件
```

**错误处理策略：**
```
API错误类型：
├─ 400 参数错误 → 提示词有问题，标记失败
├─ 401 认证失败 → API Key无效，停止生成，通知你
├─ 429 频率限制 → 等待60秒后重试（最多3次）
├─ 5xx 服务器错误 → 等待30秒后重试（最多3次）
└─ 超时 → 降级到浏览器自动化（备用方案）
```

- 5xx服务器错误：等待30秒后重试，最多3次
- 超时：降级到浏览器自动化方案

【浏览器自动化（备用方案）】
当API降级时：
1. 使用Playwright打开目标平台网页
2. 如果未登录，提示用户首次手动登录
3. 填写提示词和上传参考图
4. 点击生成并等待完成
5. 下载视频

【文件管理】
视频文件按集存放，命名规则：
{剧名缩写}_{集数}E{镜头编号}_{场景版本}_{分辨率}.mp4
每集维护一个生成状态JSON文件。

当前目标平台：{platform}
当前风格配置：
{style_config}
```

## 五、Agent 协作流程

### 5.1 数据传递方式

Agent之间通过**项目文件目录**传递数据，结构如下：

```
项目名/
├── 00_任务指令.md          ← 用户输入的任务
├── 01_故事大纲.md          ← 大纲Agent产出 → 用户审核
├── 02_完整剧情.md          ← 剧情Agent产出 → 用户审核
├── 03_分镜脚本/            ← 分镜Agent产出 → 用户审核
│   ├── 第1集.md
│   └── 第2集.md
├── 04_提示词/              ← 提示词Agent产出 → 用户审核
│   ├── 第1集.md
│   └── 第2集.md
└── 05_视频/                ← 视频Agent产出 → 用户审核
    ├── 第1集_镜头001.mp4
    └── 第2集_镜头001.mp4
```

### 5.2 审核门禁机制

```
产出文件 → [用户审核] → 通过 → 进入下一阶段
                     → 退回 → Agent修改 → 再次提交审核
```

每个阶段完成后，系统会：
1. 通知用户"XX已完成，请审核"
2. 用户审核后反馈"通过"或修改意见
3. 通过则自动进入下一阶段
4. 退回则Agent根据意见修改

### 6.1 技术选型

| 层级 | 技术 | 说明 |

```bash

├── main.py              ← 一键启动入口

### 6.3 便携性设计

- **纯Python项目**：不依赖任何IDE，有Python就能跑
- **配置外置**：API Key等敏感信息存在 `.env` 文件，不写死在代码里
- **一键安装**：`pip install -r requirements.txt` 安装所有依赖
- **跨平台**：Windows / Mac / Linux 均可运行
- **迁移方式**：复制整个项目文件夹到另一台电脑，安装Python后即可运行

## 九、补充建议（用户未提及但重要的点）

### 9.1 知识库系统
每次创作的故事设定（人物、世界观等）应该积累成知识库，后续创作时可复用或参考，避免重复设定。

### 9.2 版本控制
每个Agent的产出都保留版本记录，用户可以回退到任意历史版本，支持"试试另一种写法"。

### 9.3 模板市场
支持保存常用的故事结构模板（如"甜宠短剧模板"、"悬疑电影模板"），下次创作直接套用。

### 9.4 API Key 管理
支持多种大模型后端（OpenAI / Claude / 本地模型等），用户可自由切换。
