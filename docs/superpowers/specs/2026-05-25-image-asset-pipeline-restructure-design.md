# 图像资产管线重构设计

## 一、背景

当前系统生图管线存在以下问题：

1. **提示词未转化为实际图片** — 管线生成文字提示词后停止，未调用生图API产出 `.png` 文件
2. **角色/场景/道具提取混在一起** — 一次LLM提取包含角色+场景+道具+关联，信息混杂影响质量
3. **道具分类缺失** — 未区分场景固有道具/跨场景复用道具/角色随身道具，导致"角色带剑"还是"道具独立"不明确
4. **变体提取不稳定** — 只有一个通用prompt检测变化，容易漏掉
5. **缺少版本管理** — 多次生图结果无法对比筛选

## 二、目标

1. 建立**三层资产体系**：角色层(L1)、场景层(L2)、道具层(L3)，各自独立生成
2. 改造提取流程：从"一次提取"拆为**三步提取**（基础角色场景→道具分类→变化检测）
3. 补全生图管线：文字提示词 → 调用API → 产出实际图片
4. 增加版本管理：支持多次生成、对比、确认

## 三、三层资产体系

### 3.1 架构总览

```
第一阶段：剧本完成后 (第4阶段)

  Step 1 提取基础角色+基础场景
  Step 2 提取道具并分类
  Step 3 检测变化事件(变体)

第二阶段：资产视图生成 (半自动/自动)

  角色定妆照(空手纯色背景)  场景概念图(无人空镜)  道具展示图(独立展示)

第三阶段：逐镜头组合 (视频生成)

  角色图 + 场景图 + 道具图 + 文字提示词 → 视频
```

### 3.2 L1 — 角色资产

- **基础形象**：定妆照，单人空手，纯色背景，中性表情
- **变体形象**：基于基础形象的变化版本（受伤/换装/特定情绪）
- 规则：不持道具、全身站姿、纯色背景、不包含剧情上下文
- 用途：视频生成时的人脸参考

### 3.3 L2 — 场景资产

- **基础场景**：空境概念图，无角色、无动态事件
- **变体场景**：场景变化后的版本（爆炸后/下雨后/夜晚）
- 规则：不包含角色、只展示空间环境
- 用途：视频生成时的背景参考

### 3.4 L3 — 道具资产

- **跨场景复用道具**：需要在多个场景/场景中保持一致外观（组织标志、核心武器）；单独生成，各场景共用
- **场景固有道具**：一次性固定在某个场景中（书桌、路灯、大门）；不独立生成，写入场景描述
- **角色随身道具**：随角色移动的道具（配剑、眼镜、项链）；独立生成，角色图不带它，镜头时组合使用

## 五、提取流程改造

### 5.1 当前流程的问题

当前 `SEGMENT_EXTRACT_PROMPT` 在一个prompt中同时提取：
- 角色（含pose、手持物）
- 场景
- 道具
- 场景角色关联

导致角色描述中混入了"手持剑"等动态信息，影响定妆照生成。

### 5.2 改造方案：三步提取

将原有的一次提取拆为三步，都在第4阶段（剧本完成后）执行：

#### Step 1 — 基础角色+基础场景提取

提示词专注提取**静态信息**：

```
你是一位剧本分析师。请从以下剧本片段中提取角色和场景信息。

角色输出：name, type(main/minor), appearance, age, gender,
          clothing, accessories, key_features
⚠️ 禁止输出：pose(姿态)、手持物、动作描写

场景输出：name, environment, lighting, color_tone
⚠️ 禁止输出：动态事件、角色出现

只输出JSON。
```

#### Step 2 — 道具专项提取

```
你是一位道具分析师。请识别剧本中所有物理道具，并按类别标记。

类别规则：
- "跨场景道具"：同一道具在多个场景中出现
- "场景固有道具"：固定在某一场景中的物品
- "角色随身道具"：角色个人携带、可换场景的物品

输出JSON：[{name, category, scene(归属场景), owner(归属角色), appearance}]
```

#### Step 3 — 变化事件检测

改进当前 `CHANGE_DETECT_PROMPT`，增加精确范围约束：

```
找出角色和场景的可见状态变化事件。

角色变化：受伤/换装/发型改变/佩戴新道具
场景变化：破坏/装修/天气变化/时间推移

每个变化必须标注：
- 变化前的状态（引用基础形象）
- 变化后的可见特征
- trigger_event（触发事件，精确到场次）
- applies_from 和 applies_to（变化适用的场次范围）

只关注可见变化，不关注心理/情绪变化。
```

### 5.3 变化检测的两次LLM调用优化

当前对剧本进行了 pass1（提取）+ pass2（变化检测），其中pass2只截取了前8000字符。改进后：

- pass1 截取前6000字符（保持批次约3场的节奏）
- pass2 使用完整的剧本内容（最长12000字符），如果超长则只检测前80%的内容

## 六、道具分类决策树

```
剧本解析识别出一个道具
        │
        ▼
该道具是否固定在某个场景中？
(如：书桌、路灯、大门——不会离开该场景)
   ├─ 是 → 场景固有道具
   │        → 不独立生成图片
   │        → 写入该场景 prompt 的"道具"字段
   │
   └─ 否
        │
        ▼
该道具是否在多个场景中出现？
(如：鹰形徽章出现在基地+反派胸前+文件)
   ├─ 是 → 跨场景复用道具
   │        → 独立生成道具图片
   │        → 各场景提示词中引用该道具参考图
   │
   └─ 否
        │
        ▼
该道具是否随特定角色移动？
(如：韩信的佩剑、林雪的项链)
   ├─ 是 → 角色随身道具
   │        → 独立生成道具图片
   │        → 角色定妆照不包含该道具
   │        → 镜头描述中组合使用
   │
   └─ 否 → 一次使用道具
          → 跟场景同时生成，不单独渲染
```

## 七、变体提取与生成机制

### 7.1 变体的定义

变体 = 基础形象 + 可见变化 + 变化范围

| 变体类型 | 例子 | 触发条件 |
|:---------|:-----|:---------|
| 角色受伤 | "右臂绷带渗血" | 战斗事件 |
| 角色换装 | "换上黑色战术背心" | 场景切换/剧情节点 |
| 角色状态变化 | "脸上有泪痕" | 情感事件 |
| 场景破坏 | "天花板塌陷" | 爆炸/战斗 |
| 场景时间变化 | "变为黄昏光照" | 时间推移 |
| 场景天气变化 | "下雨，地面湿" | 天气变化 |

### 7.2 变体生成流程

```
输入：基础角色/场景 + 变化事件描述

Step 1：确定变化范围（第X场→第Y场）
Step 2：生成变体提示词
         "基于韩信誉的基础形象，在以下方面发生变化：
          - 右臂绷带包扎（有渗血）
          - 外套沾满灰尘
          变化原因：第5场战斗
          适用范围：第5场→第8场"
Step 3：调用生图API生成变体图
         参考图 = 基础形象.png
         提示词 = 基础描述 + 变化描述
Step 4：保存变体并建立基础→变体索引
```

### 7.3 变体与镜头的关联

分镜脚本中的每个镜头需要标记其所属的场次编号。通过场次编号可以确定：
- 该角色在该阶段使用哪个变体（或基础形象）
- 该场景在该阶段使用哪个变体（或基础场景）

关联逻辑：
```
对于镜头中的每个角色：
  查找该角色的变体列表
  如果变体.applies_from ≤ 镜头场次 ≤ 变体.applies_to
    使用该变体图
  否则
    使用基础形象图
```

## 八、生图管线补全

### 8.1 当前缺口

```
PromptBuilder 生成文字提示词 → 写入 .md 文件 → ❌没有调用API，没有实际图片
```

### 8.2 改造后流程

```
Step A: JSON数据 → PromptBuilder → 文字提示词
Step B: 调用 image_api.py / image_api_seedream.py → 生成 .png
Step C: 确认图片质量 → 用户可重新生成
Step D: 存入 07_生成素材/角色图|场景图|道具图 对应目录
```

### 8.3 调用策略

- 一次只生成一张图（避免并发导致API限流）
- 每张图记录生成参数（prompt、model、timestamp）
- 支持重新生成（保留历史版本v1/v2）

### 8.4 半自动模式

```
系统生成所有提示词 → 用户在前端页面浏览 → 
用户选择要生成的资产 → 点击"生成" → 逐张调用API →
用户确认或重新生成 → 确认后写入正式资产目录
```

### 8.5 全自动模式

```
可选开启：系统自动调用API生成所有资产图 →
质量检查（简单规则：无黑图、无乱码）→
自动确认并通过 → 进入视频生成阶段
```

## 十、实施计划

### 第一阶段：提取改造

1. 拆分 `SEGMENT_EXTRACT_PROMPT` 为两个独立的提示词（角色场景 + 道具）
2. 改造 `CHANGE_DETECT_PROMPT` 增强范围和格式约束
3. 修改 `VisualBibleExtractor.extract_all()` 执行三步提取
4. 修改 `VisualBibleExtractor._save_to_files()` 增加道具分类字段

### 第二阶段：生图管线

1. 新增 `07_生成素材/` 目录结构及索引管理
2. 改造 `image_preparator.py` / `image_demand_analyzer.py` 支持实际调用生图API
3. 实现半自动模式的前端交互（选择→生成→确认）
4. 实现变体与镜头的关联匹配

### 第三阶段：版本管理与优化

1. 扩展图片版本API至所有资产
2. 增加全自动模式
3. 优化变体检测准确率
4. 增加道具分类的前端展示

## 十一、风险与备注

- 生图API费用不可控，半自动模式建议默认开启
- 不同API模型的输出一致性有差异（Seedream vs 其他）
- 变体检测依赖LLM的准确性，可能需要人工复核
- 现有 `04_角色场景` 目录中的数据需迁移至新目录结构（向后兼容）
